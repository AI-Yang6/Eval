# AI Evaluation Studio — 用户场景与价值梳理

> 文档版本：v0.3 · 2026-05-27
> 阶段定位：基于当前 MVP 已实现功能，重新梳理产品解决的问题、目标用户、使用场景、成本提效价值和后续补齐方向。
> 当前状态：MVP 已开发完成。本文档从“产品构思”升级为“现有产品价值说明 + 差缺补漏清单”。

---

## 一、为什么做这个产品（Why Now）

### 1.1 行业背景

2024–2026 年，AI 应用从“能跑通 demo”进入“要稳定交付、可控迭代、可解释决策”的阶段。Prompt、模型选型、RAG 知识库和 Agent 行为已经成为 AI 产品经理的核心交付物。

但很多团队的 AI 迭代仍停留在低效状态：

- 改一版 Prompt，只跑几个样例，凭感觉判断“好像变好了”。
- 换一个模型，只看通用 benchmark，缺少业务场景下的真实对比。
- 出现 bad case 后，只在群里截图讨论，没有沉淀为可复用测试资产。
- 模型成本只在上线后才暴露，早期选型缺少“效果 / 速度 / 成本”的统一比较。
- Judge 评分和人工判断不一致时，缺少校准机制，业务方难以信任结果。

缺失的是一套面向 AI PM 的“AI 版本回归测试 + Prompt/模型 A/B 实验 + 成本评估”的轻量工作台。

### 1.2 现有方案的问题

| 方案 | 主要问题 |
|---|---|
| LangSmith / Langfuse | 更偏工程 trace 和线上观测，适合研发排查链路，不够 PM 友好；Prompt 版本、业务测试集、报告表达不是主轴 |
| PromptLayer / Helicone | 更偏调用日志和成本监控，缺少完整评估方法论和业务 case 对比 |
| 自己写 Python 脚本 | 一次性成本低，但长期维护成本高；测试集、报告、历史、回归检测难沉淀 |
| Excel + 人工打分 | 适合小样本讨论，不适合多 Prompt、多模型、多维度评估；耗时且结果难复用 |
| 只看模型官方 benchmark | 不能代表具体业务场景，也不能反映私有知识库、Prompt 约束和实际成本结构 |

### 1.3 产品切入点

AI Evaluation Studio 的切入点是：

> 面向 AI PM 和小团队，提供一个本地优先、低配置成本的 Prompt / 模型 / RAG 评估工作台，把测试集、Prompt 版本、模型配置、LLM-as-Judge、人工校准、成本估算、Bad Case 回归和报告导出打通。

## 二、目标用户（Who）

### 2.1 核心用户：AI 产品经理

**画像：**

- 负责 AI 客服、知识库问答、销售助手、情感陪伴、决策助手等 AI 功能。
- 日常工作包括写 Prompt、定义评估维度、配置模型、跟工程/算法/业务方对齐效果。
- 不一定会写脚本，但能理解测试集、版本、维度、分数、成本和报告。

**核心痛点：**

- 说不清“这版 Prompt 为什么比上一版好”。
- 模型选型缺少业务数据，只能依赖经验或供应商宣传。
- 人工验证耗时，重复测试相同 case，无法形成回归资产。
- 难以同时回答老板关心的三个问题：效果有没有提升、成本贵不贵、上线风险大不大。

### 2.2 扩展用户

- **算法 / 工程同学**：需要快速验证 Prompt、模型和 RAG 策略，不想每次手写脚本。
- **创业者 / 独立开发者**：需要低成本评估 AI 产品质量，控制模型调用费用。
- **AI 培训讲师 / 教育者**：需要演示 Prompt 迭代如何从主观感受走向数据化评估。
- **业务负责人**：不直接配置工具，但会消费导出的 Markdown / CSV 报告，用于决策。

### 2.3 非目标用户

- 已有完整 MLOps / EvalOps 平台的大型团队。
- 只需要线上 trace 监控、不做离线测试集评估的团队。
- 纯 C 端用户。
- 需要多人权限、审批流、生产级审计的企业级团队。

## 三、现有功能对应的核心场景（When / Where）

### 场景一：Prompt 版本迭代

> AI PM 修改客服 Agent 的系统提示词，希望验证新版是否真的更好。

**原来的做法：**

- 手动复制 10–30 条用户问题到模型里测试。
- 让同事主观评价哪版更好。
- 结果零散在聊天记录和表格里，下一轮还要重来。

**现在的工作流：**

1. 在测试集模块维护固定 case，支持 JSON / CSV 导入、标签筛选、分页管理。
2. 在 Prompt 模块保存 v1、v2、v3 等版本，并查看版本 diff。
3. 新建评估时选择同一测试集，对多个 Prompt 版本并行评估。
4. 通过 LLM-as-Judge 按准确性、完整性、语气、合规性等维度打分。
5. 在报告页查看雷达图、维度表、case 对比和综合排名。
6. 导出 Markdown 给老板或团队同步。

**价值：**

- 把“感觉变好了”变成“在哪些维度提升、哪些维度退化”。
- 固定测试集让每次评估可复现，减少临时抽样带来的偏差。
- Prompt 版本和报告关联，方便回看历史决策。

### 场景二：模型选型与成本决策

> 团队要在 GPT、Claude、DeepSeek、Qwen、GLM、Kimi、豆包、MiniMax 等模型中选一个线上主模型。

**原来的做法：**

- 只看官方 benchmark 或网上测评。
- 只比较效果，不系统比较速度和成本。
- 上线后才发现模型贵、慢、或某些业务 case 表现不稳定。

**现在的工作流：**

1. 在模型页配置多个 Provider 和启用模型，也可以添加自定义模型。
2. 为每个模型配置输入 / 输出 token 单价。
3. 同一评估任务里选择多个模型并行跑。
4. 报告中同时查看质量分、响应耗时、token 用量和估算成本。
5. 对比不同 Prompt × 模型组合的效果 / 成本比。

**价值：**

- 从“哪个模型最强”转为“哪个模型在我的业务里性价比最高”。
- 支持形成路由策略：低成本模型处理常规 case，高能力模型处理复杂 case。
- 在上线前预估调用成本，减少模型费用失控风险。

### 场景三：Bad Case 回归检测

> 客服 Agent 上线前发现某些退款、发票、合规问题经常答错，需要确保下一版不再犯同样错误。

**原来的做法：**

- 把 bad case 截图发群里。
- Prompt 改完后靠人工再测一遍。
- 下次类似问题又重新发现，无法沉淀。

**现在的工作流：**

1. 在报告页标记 bad case。
2. 后续评估自动检测这些 case 是否修复。
3. 报告展示已修复和仍异常的 case。
4. 历史页查看整体趋势。

**价值：**

- bad case 从“事故记录”变成“回归测试资产”。
- 降低 Prompt 改动引入新问题的风险。
- 对高风险场景形成上线前安全网。

### 场景四：RAG 知识库评估

> 团队要验证知识库问答是否能正确引用产品手册、政策文档或业务 SOP。

**原来的做法：**

- 只看单次问答效果，很难判断是模型问题、Prompt 问题还是知识库召回问题。
- Embedding 配置和模型配置混在一起，调试成本高。

**现在的工作流：**

1. 创建知识库，上传文档并切分为 chunk。
2. 配置 Embedding Provider 和模型。
3. 新建评估时关联知识库，设置 topK。
4. 每个测试 case 自动检索相关片段，并传入 Prompt。
5. 报告中保留 retrieved chunks，方便定位召回和生成问题。

**价值：**

- 将 RAG 从“看起来能问答”变成“可用固定测试集验证”。
- 帮助区分问题来源：知识没召回、Prompt 没用好、模型生成错误。
- 让知识库上线前有基本质量门槛。

### 场景五：Judge 校准与人工复核

> 业务方担心 LLM-as-Judge 不可靠，尤其是简短回答、格式约束、合规回答等场景。

**原来的做法：**

- 全人工打分太慢。
- 全自动 Judge 又缺少信任。

**现在的工作流：**

1. Judge 评分时会看到被测 System Prompt 和实际 User Prompt，理解输出约束。
2. 报告页支持人工打分。
3. 系统对比人工分和 Judge 分，展示校准差异。

**价值：**

- 保留自动化效率，同时允许业务方校准。
- 减少“模型按 Prompt 要求回答很短，却被 Judge 误判为不完整”的问题。
- 为后续优化 Judge Prompt 和评分标准提供依据。

### 场景六：数据备份与个人工作台迁移

> 个人或小团队使用本地工具，希望低成本启动，但又不能丢数据。

**现在的工作流：**

1. 数据默认存 IndexedDB，零后端数据库成本。
2. API Key 由用户本地配置。
3. 支持导出备份 JSON，包含测试集、Prompt、模型配置、评估结果、知识库等。
4. 支持恢复备份。

**价值：**

- 启动成本低，不需要部署数据库和账号系统。
- 适合作品集、个人项目、小团队内部验证。
- 通过备份机制降低浏览器本地数据丢失风险。

## 四、成本与提效价值（重点）

### 4.1 成本价值

AI Evaluation Studio 的成本价值不只是“省模型钱”，而是把模型费用、人工验证成本、试错成本和返工成本显性化。

| 成本类型 | 传统方式 | 当前工具如何降低 |
|---|---|---|
| 模型调用成本 | 多模型试错无预算意识，上线后才发现贵 | 评估阶段记录 token 用量和估算成本，提前比较不同模型的效果 / 成本比 |
| 人工评审成本 | PM / 运营 / 同事逐条手测、逐条打分 | LLM-as-Judge 自动初评，人工只复核关键 case 和校准样本 |
| 工程脚本成本 | 每轮评估都要写脚本、改参数、整理结果 | UI 化配置测试集、Prompt、模型和维度，非工程用户可独立完成 |
| 决策沟通成本 | 结果散落在聊天记录、Excel 和截图中 | 报告页 + Markdown / CSV / JSON 导出，统一对齐材料 |
| 返工成本 | Prompt 改动修一个问题又引入另一个问题 | Bad case 标记和回归检测提前发现退化 |
| RAG 调试成本 | 不知道问题来自召回、Prompt 还是模型 | 报告保留检索片段，辅助定位问题来源 |

### 4.2 提效价值

| 工作环节 | 传统耗时 | 使用工具后的变化 |
|---|---|---|
| 准备测试样本 | 临时复制问题，难复用 | 测试集长期维护，支持导入、标签和分页 |
| Prompt 版本对比 | 人工记录不同版本 | 版本保存、diff、关联评估自动沉淀 |
| 多模型测试 | 一个模型一个模型手动跑 | 多模型并行评估 |
| 打分 | 人工逐条评审 | LLM-as-Judge 批量初评，人工补充校准 |
| 报告 | 手工整理表格和截图 | 自动生成可视化报告并导出 |
| 回归验证 | 每次重新找 bad case | 标记 bad case 后自动回归检测 |

### 4.3 可量化的 ROI 口径

在对外讲述或作品集汇报时，可以用以下口径表达价值：

- **节省评估人力**：把“多人逐条试 Prompt”转为“自动批量评估 + 人工抽查”。
- **缩短迭代周期**：Prompt 从修改到拿到对比报告，可以从半天级缩短到分钟级。
- **降低模型费用风险**：上线前比较不同模型的质量、速度和估算成本，避免默认选择最贵模型。
- **降低上线风险**：历史 bad case 自动回归，减少重复犯错。
- **提升沟通效率**：报告可以直接给老板、业务方或工程同学看，减少主观争论。

### 4.4 成本决策示例

假设一个客服场景有 500 万次月调用：

- 高能力模型单次成本高，但在复杂 case 上准确率更好。
- 低成本模型单次便宜，但在部分复杂 case 上更容易出错。

通过 AI Evaluation Studio 可以先在离线测试集里得到：

- 常规问题：低成本模型得分接近高能力模型。
- 复杂合规问题：高能力模型明显领先。
- 成本估算：如果全部走高能力模型，月成本显著上升。

因此可以形成策略：

- 80% 常规流量走低成本模型。
- 20% 高风险 / 复杂 case 路由到高能力模型。
- 用固定测试集持续验证路由策略是否影响质量。

这就是工具的核心成本价值：不是盲目省钱，而是在质量可控的前提下降低单位调用成本。

## 五、价值主张（What's the Value）

### 5.1 一句话价值

> 让 AI PM 像看 A/B 实验报告一样，评估 Prompt、模型和知识库策略的质量、成本和风险。

### 5.2 四层价值

| 层级 | 价值 | 对应功能 |
|---|---|---|
| 效率层 | 自动化多 Prompt、多模型、多 case 评估，减少人工试错 | 测试集、并行评估、LLM-as-Judge |
| 成本层 | 评估效果 / 速度 / token 成本，支持性价比选型 | 模型价格、token usage、成本估算、报告导出 |
| 决策层 | 把主观争论转成可复现数据，支持团队对齐 | 雷达图、维度表、综合排名、Markdown / CSV 导出 |
| 风险层 | 通过 bad case 回归、人工校准、备份恢复降低上线和数据风险 | Bad case、回归检测、人工评分、Judge 校准、备份恢复 |

### 5.3 差异化

- **比 Excel 更自动化**：支持批量评估、自动评分、图表报告、历史趋势。
- **比脚本更产品化**：测试集、Prompt 版本、模型配置、报告和备份都可视化。
- **比 trace 工具更 PM 友好**：不是从工程日志出发，而是从“业务测试集 + 评估报告”出发。
- **比通用 benchmark 更贴业务**：使用用户自己的 case、Prompt、知识库和评分维度。

## 七、差缺补漏与后续优化方向

### 7.1 成本提效方向

当前已有成本估算，但还可以继续增强：

1. **成本预算栏**
   - 新建评估前预估本次评估可能花费多少。
   - 超过预算时提示用户缩小测试集、减少模型或降低 topK。

2. **效果 / 成本比指标**
   - 在报告页新增 `综合分 / $1` 或 `每 1 美元可获得的有效得分`。
   - 帮助用户直接识别高性价比模型。

3. **模型路由建议**
   - 根据 case 难度、分数差异和成本差异，建议“常规 case 用低成本模型，复杂 case 用高能力模型”。

4. **失败调用成本提示**
   - 当前失败 case 会记录 error，但可以进一步统计失败率。
   - 失败率高的模型即使单价便宜，也可能带来返工成本。

5. **评估任务规模预估**
   - 在新建评估页展示组合数量：`测试用例数 × Prompt 版本数 × 模型数`。
   - 提醒用户评估规模和预计耗时。

### 7.2 评估可信度方向

1. **Judge 多次采样 / 多 Judge 交叉验证**
   - 对高风险场景可让 Judge 跑多次取均值。
   - 或使用两个 Judge 模型交叉验证。

2. **Rubric 模板升级**
   - 按场景提供模板：客服、销售、RAG、合规、情感陪伴、分类判断。
   - 对“只回答是/否”“只输出 JSON”等格式约束提供专门维度。

3. **人工校准样本集**
   - 将人工评分过的 case 作为 Judge 校准集。
   - 长期提升 Judge prompt 的稳定性。

### 7.3 工程与产品化方向

1. **评估任务持久化执行**
   - 当前评估由前端会话触发，刷新会影响内存中的取消状态。
   - 后续可改为后端任务队列或本地 worker。

2. **团队协作**
   - 支持共享测试集、共享报告、多人评分。

3. **数据安全**
   - 当前是本地优先，适合个人和可信环境。
   - 若部署公网，需要补鉴权、权限、密钥管理和服务端持久化。

4. **报告增强**
   - 增加 PNG / PDF 导出。
   - 增加“结论摘要”和“下一步建议”自动生成。

## 八、产品定位与边界

### 8.1 一句话定位

**AI PM 的 Prompt、模型和 RAG 评估工作台。**

它不是线上 trace 监控平台，不是 Agent 编排平台，也不是单纯的 Prompt 编辑器，而是围绕“离线测试集评估、版本对比、成本决策、回归验证”展开。

### 8.2 MVP 明确不做

- 不做生产级多人权限和审计。
- 不做线上 trace / session replay。
- 不做 Agent 流程编排。
- 不做自动上线和线上流量分配。
- 不承诺模型价格实时准确，当前价格用于估算和决策参考。

### 8.3 当前已超出最初 MVP 的能力

- RAG 知识库和 Embedding 统一配置。
- 人工评分与 Judge 校准。
- Bad case 标记和回归检测。
- 成本估算。
- 数据备份与恢复。
- 报告导出。
- 多 Provider 和自定义模型。
