## 二、各模块详细拆解

### 模块 1：测试集管理

**对应场景**：场景一（prompt 迭代）、场景二（模型选型）、场景三（bad case 回归）

**核心能力**：
| 功能 | 描述 | 优先级 |
|---|---|---|
| 创建测试集 | 填写名称 + 描述，选择类型（对话/单轮/结构化） | P0 |
| 添加测试用例 | 逐条添加：输入(input) + 期望输出(expected) + 标签(tag) | P0 |
| 批量导入 | 支持 JSON/CSV 文件上传，自动解析 | P0 |
| 测试集列表 | 展示所有测试集，显示用例数量、最近使用时间 | P0 |
| 编辑/删除 | 编辑用例内容，删除单条或整个测试集 | P0 |
| 标签筛选 | 按 tag 筛选测试用例（如"退款流程"、"问候语"） | P1 |

**MVP 数据结构**：
```
TestSuite {
  id, name, description, type, createdAt, updatedAt
  cases: TestCase[]
}

TestCase {
  id, input, expected, tags[], actualOutputs: {
    [runId]: string
  }
}
```

**设计原则**：
- 测试集是整个工作流的起点，必须最简洁 — 能 30 秒内创建一个测试集
- 支持最简形式：只有 input，expected 可选（因为很多场景很难定义"标准答案"）

---

### 模块 2：Prompt 版本管理

**对应场景**：场景一（prompt 迭代）、场景三（bad case 回归）

**核心能力**：
| 功能 | 描述 | 优先级 |
|---|---|---|
| 创建 Prompt | 填写名称，输入 system prompt + user prompt 模板 | P0 |
| 版本记录 | 每次编辑自动存为新版本，可查看 diff | P0 |
| 变量支持 | 用 `{{variable}}` 语法，运行时填入 | P1 |
| 版本对比 | 两个版本并排 diff 展示 | P1 |
| 版本列表 | 展示所有版本，标注哪个版本跑过评估、得分多少 | P0 |

**MVP 数据结构**：
```
Prompt {
  id, name, createdAt
  versions: PromptVersion[]
}

PromptVersion {
  id, versionNumber, systemPrompt, userPromptTemplate,
  createdAt, evalRuns: EvalRun[] (关联的评估记录)
}
```

**设计原则**：
- MVP 不做复杂变量系统，只支持 `{{input}}` 一个内置变量（映射测试集的 input）
- 版本号自动递增，用户只需"保存为新版本"
- Diff 用最简单的文本对比，不做语法级别高亮

---

### 模块 3：模型配置与选型

**对应场景**：场景二（模型选型决策）

**核心能力**：
| 功能 | 描述 | 优先级 |
|---|---|---|
| 添加模型 | 选择 Provider（OpenAI / Anthropic / DeepSeek / 自定义），填入 API Key | P0 |
| 模型列表 | 展示已配置模型，显示 provider、模型名、单 token 成本 | P0 |
| 模型对比选择 | 创建评估任务时，可勾选多个模型并行跑 | P0 |
| 成本估算 | 根据测试集大小和模型价格，估算本次评估成本 | P1 |

**MVP 支持的 Provider**：
| Provider | 模型 | 优先级 |
|---|---|---|
| OpenAI | GPT-4o, GPT-4o-mini | P0 |
| Anthropic | Claude Sonnet 4.6, Claude Haiku 4.5 | P0 |
| DeepSeek | DeepSeek-V3, DeepSeek-Chat | P1 |
| 自定义 (OpenAI 兼容) | 用户填 base URL + model name | P1 |

**设计原则**：
- API Key 存 localStorage，MVP 不做后端存储
- Provider 用适配器模式，统一调用接口，方便后续扩展
- 成本估算是加分项，先做粗粒度（按输入 token 数 × 单价）

### 模块 4：评估任务引擎（核心模块）

**对应场景**：所有场景的核心执行层

**核心能力**：
| 功能 | 描述 | 优先级 |
|---|---|---|
| 创建评估任务 | 选择：测试集 + prompt 版本(1或多个) + 模型(1或多个) + 评估维度 | P0 |
| 并行执行 | 多个 prompt × 多个模型组合，并行调用 API | P0 |
| 实时进度 | 显示每个组合的执行进度（n/total） | P0 |
| LLM-as-Judge | 用一个模型对输出按 rubric 打分 | P0 |
| 人工打分 | 展示输出，让用户逐条打 1–5 分 | P1 |
| 错误处理 | API 调用失败时记录错误，不阻塞其他任务 | P0 |

**评估任务的数据流**：
```
1. 用户选择: 测试集(30条) × Prompt(v2, v3) × 模型(Claude, GPT-4o)
   → 4 个组合，每个组合 30 次调用 = 120 次 API 调用

2. 执行引擎并行跑 4 个组合
   → 每个 case 的输出存入 actualOutputs

3. LLM-as-Judge 对每个输出按评估维度打分
   → 评分结果存入 evalScores

4. 汇总报告
```

**LLM-as-Judge 的设计**：
```
JudgePrompt = `
你是一个评估助手。请根据以下维度对 AI 回复打分（1-5分）。

评估维度：
{{rubric}}

用户输入：{{input}}
期望输出方向：{{expected}}
AI 实际回复：{{actual}}

请按每个维度打分并给出简短理由，输出 JSON：
{"scores": {"维度1": 3, "维度2": 4}, "reasoning": "..."}
`
```

**MVP 数据结构**：
```
EvalRun {
  id, name, createdAt, status: 'running' | 'completed' | 'failed'
  testSuiteId, promptVersions[], models[], rubric[]
  results: EvalResult[]
}

EvalResult {
  testCaseId, promptVersionId, modelId
  actualOutput, tokenUsage, latency
  scores: { [dimension]: number }
  judgeReasoning: string
}
```

**设计原则**：
- 核心体验：用户点"运行"后能看到实时进度条，不是干等
- 并发控制：前端控制并发数（默认 3），避免 rate limit
- Judge 模型默认用最便宜的高能力模型（Haiku / GPT-4o-mini）

### 模块 5：对比报告与可视化

**对应场景**：场景一（prompt 迭代对比）、场景二（模型选型矩阵）

**核心能力**：
| 功能 | 描述 | 优先级 |
|---|---|---|
| 维度得分雷达图 | 每个组合的各维度得分，用雷达图对比 | P0 |
| 版本/模型对比矩阵 | 行=测试用例，列=组合，格子=得分，高亮最优 | P0 |
| 单 case 详情 | 点击某个格子，看具体输出 + judge 评语 | P0 |
| 综合得分排名 | 各组合的综合分排名 | P0 |
| 成本对比 | 各组合的 token 消耗 + 估算成本 | P1 |
| 导出报告 | 导出为 PDF / PNG / JSON | P2 |

**报告布局草图**：
```
┌────────────────────────────────────────────┐
│  评估报告：客服 prompt v2 vs v3              │
│  测试集：30条 | 模型：Claude Sonnet 4.6      │
│  时间：2026-05-25                            │
├────────────────────────────────────────────┤
│                                            │
│   📊 综合得分                                │
│   v2: 3.8 / 5    v3: 4.1 / 5   ↑ +0.3     │
│                                            │
├────────────────────────────────────────────┤
│                                            │
│   🕸️ 雷达图（维度对比）                       │
│      亲切度  专业度  完整度  准确度             │
│   v2   2.5    4.2    3.8    4.0             │
│   v3   4.1    3.5    3.2    3.9             │
│                                            │
├────────────────────────────────────────────┤
│                                            │
│   📋 逐 case 对比矩阵                        │
│   #1  v2:4  v3:5  │ #2  v2:3  v3:3        │
│   #3  v2:5  v3:4  │ #4  v2:2  v3:4        │
│   ... (可展开看详情)                          │
│                                            │
└────────────────────────────────────────────┘
```

**设计原则**：
- 报告是产品最有力的"wow moment"，必须一打开就让人觉得"这比 Excel 强太多"
- 雷达图用 recharts（Next.js 生态内最好用）
- MVP 不做 PDF 导出，先做页面内可截图

### 模块 6：迭代历史与回归

**对应场景**：场景三（bad case 回归）

**核心能力**：
| 功能 | 描述 | 优先级 |
|---|---|---|
| 评估历史列表 | 展示所有历史评估，按时间倒序 | P0 |
| 版本趋势图 | 同一 prompt 的各版本得分趋势折线图 | P0 |
| Bad case 收集 | 标记某个 case 为 bad case，自动加入"回归测试集" | P1 |
| 回归检测 | 新评估自动检查：之前标记的 bad case 是否修复（报告页展示对比表） | P1 |
| 对比任意两次评估 | 选两次评估，看 diff | P1 |

**设计原则**：
- 迭代历史是"资产沉淀"的核心 — 让用户越用越离不开
- MVP 先做历史列表 + 趋势图，bad case 回归放 P1

---

### 数据层

**MVP 方案：localStorage + IndexedDB**

| 数据 | 存储方式 | 原因 |
|---|---|---|
| 用户配置（API Key 等） | localStorage | 简单，MVP 够用 |
| 测试集 | IndexedDB | 可能较大（JSON 文件导入） |
| Prompt + 版本 | IndexedDB | 版本多时数据量大 |
| 评估结果 | IndexedDB | 结果数据量大 |
| 评估报告摘要 | IndexedDB | 需要离线访问 |

**远期迁方案**：Supabase（免费 tier 足够 demo 用）

## 四、用户主流程（串联所有模块）

```
Step 1: 准备
  → 创建测试集（手写 or 导入）
  → 写 Prompt，保存为 v1
  → 配置模型（填 API Key）

Step 2: 运行评估
  → 选测试集 + Prompt v1 + 模型 → 点"运行"
  → 看实时进度条
  → LLM-as-Judge 自动打分

Step 3: 看报告
  → 雷达图看各维度
  → 点进具体 case 看输出和评语

Step 4: 迭代
  → 改 Prompt，保存为 v2
  → 再跑一次评估
  → 对比 v1 vs v2 的得分变化
  → 看趋势图：迭代方向对不对

Step 5: 回归（P1）
  → 标记 bad case
  → 下次评估自动检查是否修复
```

## 五、技术架构概览（为 PRD 和 MVP 做铺垫）

```
┌─────────────────────────────────────────┐
│              Next.js App Router          │
│                                         │
│  /test-suites    测试集管理页             │
│  /prompts        Prompt 管理页           │
│  /models         模型配置页              │
│  /evaluations    评估任务页              │
│  /reports/[id]   对比报告页              │
│  /history        迭代历史页              │
│                                         │
├─────────────────────────────────────────┤
│            核心业务逻辑层                 │
│                                         │
│  eval-engine    并行执行 + Judge 逻辑    │
│  model-adapter  统一多模型调用接口        │
│  scoring        评分计算 + 聚合          │
│                                         │
├─────────────────────────────────────────┤
│              数据层                      │
│  localStorage   用户配置                 │
│  IndexedDB      业务数据                 │
│  (via Dexie.js)                         │
└─────────────────────────────────────────┘
```

**关键技术选型**：
| 技术 | 用途 | 选型理由 |
|---|---|---|
| Next.js 14+ (App Router) | 框架 | 社区生态好，Vercel 一键部署 |
| Tailwind CSS + shadcn/ui | UI | 快、好看、可定制 |
| recharts | 可视化 | React 生态内最成熟的图表库 |
| Dexie.js | IndexedDB 封装 | 比 raw IndexedDB 好用 10 倍 |
| zustand | 状态管理 | 轻量，MVP 够用 |
| Vercel AI SDK | 模型调用 | 统一多 provider 调用接口，内置流式 |
